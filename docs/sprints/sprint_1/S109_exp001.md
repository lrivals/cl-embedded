### 2. Boucle CL principale

```python
def train_ewc(model, tasks, config, device):
    import numpy as np
    import torch
    import torch.nn as nn

    T = len(tasks)
    acc_matrix = np.full((T, T), np.nan)

    epochs = config["training"]["epochs_per_task"]
    lr = config["training"]["learning_rate"]
    momentum = config["training"]["momentum"]
    ewc_lambda = config["ewc"]["lambda"]
    gamma = config["ewc"]["gamma"]
    n_fisher_samples = config["ewc"]["n_fisher_samples"]

    optimizer = torch.optim.SGD(model.parameters(), lr=lr, momentum=momentum)

    fisher = None      # None pour Task 1 (pas de régularisation)
    theta_star = None  # None pour Task 1

    model.to(device)

    for i, task in enumerate(tasks):
        model.train()

        for _ in range(epochs):
            for x, y in task["train_loader"]:
                x, y = x.to(device), y.to(device)
                optimizer.zero_grad()
                loss = model.ewc_loss(x, y, fisher, theta_star, ewc_lambda)
                loss.backward()
                optimizer.step()

        # Fin de la tâche : snapshot θ* + mise à jour Fisher Online
        theta_star = model.get_theta_star()
        new_fisher = compute_fisher_diagonal(
            model, task["train_loader"], device, n_samples=n_fisher_samples
        )
        fisher = update_online_fisher(fisher, new_fisher, gamma=gamma)

        # Évaluation sur toutes les tâches vues
        model.eval()
        for j in range(i + 1):
            from src.evaluation.memory_profiler import profile_forward_pass
            from src.training.baselines import evaluate_task
            acc_matrix[i, j] = evaluate_task(model, tasks[j]["val_loader"], device)

    return acc_matrix
```

### Analyse et commentaires

**Points positifs :**

1. **Critère principal validé** : `aa_ewc (0.9824) > aa_naive (0.9811)` ✅
2. **Contrainte matérielle respectée** : RAM peak update = 6.8 Ko, soit **10.4% du budget 64 Ko**. Large marge pour le portage STM32N6. ✅
3. **Latence bien en-dessous de la cible** : 0.036 ms vs cible ≤ 100 ms. ✅
4. **Régularisation EWC efficace** : AF = 0.001, quasi-nul. Le modèle ne subit pas d'oubli catastrophique.

**Observations importantes :**

5. **Oubli catastrophique quasi-absent sur les deux méthodes** : le fine-tuning naïf atteint lui aussi AF ≈ 0. Ceci suggère que les 3 domaines du Dataset 2 (Pump/Turbine/Compressor) partagent des distributions de features très proches et ne présentent pas de *domain shift* fort. C'est une **limitation pour valider l'efficacité différentielle d'EWC** sur ce dataset.

6. **Joint training ≤ EWC sur la AA finale** : résultat contre-intuitif (la borne supérieure théorique devrait être `aa_joint ≥ aa_ewc`). Explication probable : l'optimisation séquentielle avec SGD converge vers un meilleur minimum par tâche que l'entraînement joint sur 3 domaines mélangés. Phénomène de **curriculum implicite**.

7. **Nombre de paramètres = 705** (et non 769 estimé dans les specs). Architecture 4→32→16→1 : calcul exact = (4×32+32) + (32×16+16) + (16×1+1) = 160 + 528 + 17 = 705 paramètres.

**Implications pour le manuscrit :**

- `FIXME(gap1)` : le Dataset 2 ne montre pas de domain shift significatif. Pour combler Gap 1 (validation sur données industrielles avec drift réel), il faudra soit montrer des résultats sur Dataset 1 (Pump, temporel), soit utiliser un protocole de split plus agressif sur Dataset 2.
- `FIXME(gap2)` : **Gap 2 est en bonne voie** — RAM peak update mesurée = 6.8 Ko, très en dessous des 64 Ko. Chiffres à reporter dans la Table Gap 2 (Section 4.3 du manuscrit).
- La quasi-absence d'oubli catastrophique constitue en soi un résultat : ce dataset est trop facile pour EWC. À nuancer dans le manuscrit.
